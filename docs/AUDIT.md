## 2. Целостность: расхождения «док vs код» (критично)

`docs/HANDOFF.md` описывает желаемое как сделанное. Проверено grep'ом — **в коде отсутствует**:

- **Банкротство §3.10:** заявлены статусы `at_risk`/`liquidated`, grace-период, распродажа активов банкрота лотами. В коде только `ACTIVE`/`BANKRUPT`, ничего из этого нет (0 совпадений).
- **Чат и переговоры §4.7:** `ChatMessage`, `Negotiation`, офферы — нет (0).
- **События §4.8:** погода, праздники, урожайность, поломки — нет (есть только абстрактные «рыночные события» + сезонность).
- **Контракты §4.4:** SLA, отсрочка платежа, частичное исполнение, арбитраж — нет.

**Вывод:** канонический дизайн-док врёт о статусе. `DEVELOPMENT_STATUS.md` точен (ведётся по факту). Действие: HANDOFF разжаловать из «что сделано» в «исходное широкое видение», статус-правда живёт в `DEVELOPMENT_STATUS.md`, дизайн-правда — в переписанном `TECHNICAL_SPEC.md`.

## 3. Главная дизайн-проблема: цепочка поставок почти полая

Игра называется «Цепочка прибыли», жанр — про supply chain. Но:

- **30 товаров, 3 сырья, 3 рецепта.** Реально производятся только хлеб и молоко.
- Остальные **27 товаров не имеют производственного пути** — они появляются только через стартовые остатки и NPC-рынок «из воздуха».
- Значит ядро фантазии («построй и оптимизируй цепочку сырьё→производство→логистика→полка») **не существует для ~90% каталога**.

Это разрыв №1: без него обещание жанра пустое. Расширение рецептур/сырья — самая ценная инвестиция (см. ROADMAP, Фаза 1).

## 4. Экономика: где механики «недокручены»

- **Нет эластичности спроса.** Региональный спрос фиксирован: `total_demand = base × demand_index × event × season`. Цена влияет только на *долю* через `вес = демонстратор × маркетинг / price_index`. → Снижение цены не растит рынок, только перетягивает чужую долю (чистый zero-sum). Хук «ценовые войны» из-за этого плоский.
- **Мёртвые параметры регионов.** `logistics_risk` и `income_index` заданы в 5 регионах, но **нигде не читаются движком** (0 совпадений). Региональная дифференциация наполовину декоративна.
- **Нет сегментов покупателей, замещения товаров, явного lost-sales** (кроме упора в мощность магазина).
- **Логистика тонкая.** Заявки на доставку есть, но маршрутов, времени/риска перевозки, cold chain, консолидации — нет. Роль дистрибьютора недонаполнена.
- **NPC примитивны.** 3 стратегии задают лишь `price_index` + маркетинг. Не строят цепочки, не реагируют на дефицит, не расширяются, не торгуют. Для выбранного «одиночка против умных ботов» это **главный геймплейный долг**.

## 5. Архитектура: оценка и риски

**Сильные стороны:** чистое разделение `models / balance / engine / api`, Pydantic-контракты, 200 тестов, идемпотентность дня, понятный баланс вынесен в `balance.py`.

**Риски и долги:**

1. **Два параллельных мира.** Глобальные `_state/_engine/_auth` (основная партия) + `SessionRegistry` (мультиплеер). Эндпоинты раздвоены, логика дублируется. Нужно унифицировать: основная партия = сессия по умолчанию.
2. **Хранилище — снапшот, а не модель.** Даже в «DB-режиме» весь мир — один JSON-blob в одной строке. Это блокирует конкурентные операции, аналитику и поиск. **НО:** для выбранного «одиночка против ботов» нормализация в PostgreSQL **не нужна в ближайшее время** — ROLE.md/HANDOFF её переоценивают. Откладываем.
3. **`engine.py` — god-class** (1600 строк, 73 метода). `close_day` дёргает ~12 подсистем последовательно. Работает и протестирован, но будет сопротивляться росту. Со временем — разнести подсистемы по модулям (`engine/retail.py`, `engine/production.py`, …) за общим оркестратором.
4. **Дневной цикл не транзакционен** в DB-режиме (мутируем in-memory, потом дампим blob). Для одиночной игры приемлемо; для мультиплеера — нет.
5. **Нет харнесса баланса.** `simulation.py` — 32 строки. Нельзя автоматически измерить «здоровье экономики» (инфляция, разорения, доминирование). Нужен прогон на N дней с метриками — это инструмент гейм-дизайна.

## 6. Игровые кейсы: где путь игрока ломается

| Кейс | Статус | Где ломается |
|---|---|---|
| Онбординг нового игрока | 🔴 | После «создать компанию» — стена из 15 панелей, нет туториала, неясно «что делать». |
| Ритейлер: построй→затарь→цена→продай | 🟡 | Работает, но «откуда товар» неочевидно; нет понятного поиска поставщика. |
| Производитель: сырьё→продукт | 🔴 | Производимы только хлеб/молоко (3 рецепта). Ядро роли пустое. |
| Дистрибьютор: вози и зарабатывай | 🟡 | Заявки есть, но без маршрутов/риска/cold chain — роль скучна. |
| Победа / поражение | 🟢 | Победа при кэше ≥100 млн или последний выживший; банкрот заморожен. Порог произволен, скоринг плоский. |
| Игра с друзьями | 🔴 | Сессии есть в API, но нет лобби/UI и чата — реально не сыграть. |
| Продолжить после рестарта | 🟢 | Состояние и зал славы персистятся (JSON/DB). |

## 7. Сводка приоритетов (детали — в ROADMAP.md)

1. **Целостность:** привести доки к коду (убрать «сделано», которого нет). *(дёшево, сразу)*
2. **Наполнить цепочку:** ↑ рецептов и сырья до играбельного набора (≈12–15 производимых товаров по категориям). *(ядро жанра)*
3. **Оживить спрос:** эластичность + простые сегменты + lost-sales + задействовать `income_index`. *(хук «цены»)*
4. **Умные боты:** NPC строят цепочки, реагируют на дефицит/цены, расширяются. *(это и есть одиночная игра)*
5. **Логистика:** маршруты, время/риск перевозки, cold chain, склад как буфер; задействовать `logistics_risk`. *(хук «оптимизация»)*
6. **Игровой UI:** карта-центр, экраны ролей, туториал, советник/алерты. *(читаемость)*
7. **Контракты/торговля:** регулярные поставки, отсрочка, SLA, NPC-переговоры, репутация. *(хук «торговля»)*
8. **События/сезонность, рост империи, и позже — мультиплеер + нормализация БД.** *(полировка/масштаб)*

**Инструмент на всё:** харнесс баланса (прогон симуляции + метрики экономики) — строим в Фазе 0, используем как «тест на веселье/баланс» далее.
