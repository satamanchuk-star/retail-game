# План действий для команды разработки

Этот план переводит большое ТЗ в управляемые инженерные этапы. Каждый этап должен завершаться тестами, обновлением документации, коммитом и PR.

## Принципы выполнения

1. Один PR — один понятный слой.
2. Не ломать существующие API без миграционного периода.
3. Любая новая доменная логика должна иметь тест.
4. Любое новое состояние должно быть видно через API или UI.
5. Секреты только через окружение.
6. Сначала устойчивость ядра, потом глубина геймплея.

## Этап 0. Вход в проект

**Цель:** разработчик понимает текущий контекст.

**Действия:**

1. Прочитать `README.md`.
2. Прочитать `ROLE.md`.
3. Прочитать `docs/HANDOFF.md`.
4. Прочитать `docs/TECHNICAL_SPEC.md`.
5. Запустить `pytest -q`.
6. Запустить приложение локально.
7. Проверить `/api/project/status`.

**Критерий готовности:** разработчик может объяснить текущую стадию и следующий этап.

## Этап 1. Нормализованная PostgreSQL persistence

**Цель:** заменить snapshot на табличную модель.

**Задачи:**

1. Добавить Alembic.
2. Создать ORM-модели для пользователей, компаний, регионов, товаров, сырья, рецептов, партий, объектов, контрактов, кредитов, ledger, отчётов.
3. Сделать seed стартового мира.
4. Перевести чтение/запись ключевых сущностей на репозитории.
5. Сохранить snapshot fallback только как dev/demo режим.

**Проверки:**

- `pytest -q`;
- интеграционный тест создания компании и закрытия дня через PostgreSQL/SQLite;
- перезапуск сервера сохраняет компании, кредиты, контракты и день.

## Этап 2. Транзакционное закрытие дня

**Цель:** закрытие дня должно быть атомарным.

**Задачи:**

1. Вынести расчёт дня в сервис `DayClosureService`.
2. Создать таблицу закрытий дня со статусами `pending/running/completed/failed`.
3. Использовать `closure_id` как ключ идемпотентности.
4. Сделать блокировку или optimistic concurrency для одного мира.
5. Писать audit-log операций.

**Проверки:**

- повторный `closure_id` не меняет состояние;
- ошибка в середине расчёта откатывает транзакцию;
- параллельные запросы не закрывают один день дважды.

## Этап 3. Улучшение auth

**Цель:** безопаснее управлять пользователями.

**Задачи:**

1. Ввести TTL сессии или JWT.
2. Добавить logout.
3. Добавить rate-limit для регистрации/логина.
4. Добавить смену пароля позже, если понадобится.

**Проверки:**

- чужой пользователь не управляет компанией;
- истёкшая сессия не работает;
- `/api/state` не раскрывает секреты.

## Этап 4. Ритейл 1.0

**Цель:** магазин становится полноценной моделью.

**Задачи:**

1. Добавить форматы магазинов.
2. Добавить площадь, полки, склад, холодильники.
3. Добавить персонал и аренду.
4. Добавить цены по SKU.
5. Добавить промо базового уровня.
6. Добавить спрос по магазину.

**Проверки:**

- магазин не продаёт больше мощности;
- отсутствие товара создаёт lost sales;
- промо меняет спрос и маржу.

## Этап 5. Производство 1.0

**Цель:** производитель управляет линиями и партиями.

**Задачи:**

1. Добавить производственные линии.
2. Добавить больше рецептов для 30 товаров.
3. Добавить срок производства.
4. Добавить качество и брак.
5. Добавить закупку сырья у NPC.

**Проверки:**

- нельзя произвести без сырья;
- партия появляется после срока производства;
- качество влияет на спрос/цену.

## Этап 6. Дистрибуция 1.0

**Цель:** логистика становится самостоятельной ролью.

**Задачи:**

1. Добавить транспорт.
2. Добавить маршруты.
3. Добавить склады с типами хранения.
4. Добавить cold chain.
5. Добавить задержки и повреждения.
6. Добавить перепродажу товара с баланса.

**Проверки:**

- нельзя перевезти больше вместимости;
- холодильный товар портится без cold chain;
- задержка влияет на SLA.

## Этап 7. Контракты 1.0

**Цель:** контракты становятся главным рынком игры.

**Задачи:**

1. Добавить типы: регулярные, SLA, хранение, отсрочка, промо, эксклюзив.
2. Добавить частичное исполнение.
3. Добавить штрафы и долги.
4. Добавить репутацию контрагентов.
5. Добавить арбитраж.

**Проверки:**

- штраф списывается или становится долгом;
- форс-мажор может снизить штраф;
- нарушение влияет на репутацию.

## Этап 8. Боты 1.0

**Цель:** рынок живёт без полного состава игроков.

**Задачи:**

1. Добавить стратегии ботов.
2. Боты принимают решения каждый день.
3. Боты заключают контракты.
4. Боты берут кредиты и банкротятся.

**Проверки:**

- бот создаёт дневное решение;
- бот может выполнить контракт;
- сложность влияет на эффективность.

## Этап 9. Чат и переговоры

**Цель:** игроки договариваются внутри игры.

**Задачи:**

1. Общий чат.
2. Приватный чат.
3. Групповые переговоры.
4. Офферы.
5. Принятие оффера в контракт.
6. Жалобы и модерация.

**Проверки:**

- приватный чат доступен только участникам;
- оффер создаёт контракт после принятия;
- жалоба сохраняется.

## Этап 10. События и сезонность

**Цель:** рынок реагирует на внешний мир.

**Задачи:**

1. Времена года.
2. Погода.
3. Региональные события.
4. Глобальные события.
5. Прогнозы на 1–3 дня.

**Проверки:**

- снег увеличивает время доставки;
- жара ускоряет порчу;
- праздник повышает спрос.

## Этап 11. UI 1.0

**Цель:** уйти от демо-страницы к настоящему игровому интерфейсу.

**Задачи:**

1. Навигация по экранам.
2. Карта.
3. Ролевые экраны.
4. Финансовый экран.
5. Контракты и переговоры.
6. Графики.
7. Уведомления и советник.

**Проверки:**

- пользователь может пройти основной путь без обращения к API вручную;
- e2e smoke: register → company → decision → close day → report.

## Этап 12. Админка баланса

**Цель:** настраивать экономику без изменения кода.

**Задачи:**

1. Admin API.
2. Таблицы баланса.
3. История изменений.
4. Метрики экономики.
5. Dev-only UI.

**Проверки:**

- изменение баланса влияет только на будущие дни;
- история изменений сохраняется;
- нет доступа без admin-прав.

## Этап 13. Подготовка закрытой альфы

**Цель:** дать пользователю поиграть самому и с друзьями.

**Задачи:**

1. Стабилизировать deployment.
2. Добавить seed/fixtures для тестового мира.
3. Добавить инструкции запуска.
4. Добавить мониторинг ошибок.
5. Провести балансировочный прогон на несколько игровых недель.

**Критерий:** можно провести тестовую игру с несколькими пользователями без ручного вмешательства разработчика.
